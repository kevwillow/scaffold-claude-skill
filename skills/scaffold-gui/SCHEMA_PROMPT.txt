=== TYPES ===

Type       | Use when                                    | Wrong names → correct
-----------|---------------------------------------------|----------------------
boolean    | Flag takes no value, set separator to "none" | "bool" → "boolean"
string     | Free-form single-line text                  | —
text       | Multi-line input (rare)                     | "textarea" → "text"
integer    | Whole number value                          | "int","number" → "integer"
float      | Decimal number value                        | "num" → "float"
enum       | ONE value from a STRICTLY CLOSED set        | "select","dropdown" → "enum"
multi_enum | MULTIPLE values from a closed set           | "list" → "multi_enum"
file       | File path (only when docs clearly say file) | "filepath","path" → "file"
directory  | Directory path (only when docs say dir)     | "filename" → use "file"
password   | Sensitive value (masked in GUI)             | —

Use ONLY these types. When unsure between file and string, use string. When unsure between password and string, use string.

=== RULES ===

1. POSITIONAL: Set positional to true. Use a label like "HOST" for flag. Positional args MUST be the LAST entries in the array — the GUI appends them after all flags.

2. MUTUAL EXCLUSIVITY: Docs show --flagA | --flagB or "cannot be used with" → assign the same group string.

3. SEPARATOR: Look for "=" in docs. "--flag=VALUE" → "equals". "--flag VALUE" → "space". Boolean or concatenated like -T4 → "none". "none" means flag and value join directly: -T4 not -T 4.

4. EXAMPLES vs ENUM: enum locks the GUI to ONLY those values — use it only when no other value is valid. examples shows suggestions but the user can type anything. Use examples when docs list common values but the flag accepts others too (phrases like "see list", "supported values include").

5. SUBCOMMANDS: Only for clearly subcommand-based tools (git, docker, ansible-galaxy). Most tools → null. Global flags go in top-level arguments, subcommand flags go in each subcommand's arguments. For tools with multi-level subcommands (e.g. "ansible-galaxy role install", "docker compose up"), use the full subcommand chain as the name: "role install", "compose up". Each word becomes a separate token in the assembled command.

6. ELEVATED: "optional" if some features need root (e.g. nmap SYN scan). "always" if tool requires root (e.g. iptables). null when in doubt.

7. DEFAULTS: Only when docs explicitly state them. Use correct JSON type.

8. COMPLETENESS: Include ALL documented flags, regardless of how many there are. If output is truncated, the user will re-prompt to continue.

9. ACCURACY: Only include flags from the provided documentation. Do not invent flags.

10. NO DUPLICATES: Short + long forms of the same flag = ONE entry (long in "flag", short in "short_flag"). Aliases → pick the primary name, omit the alias.

11. UNCLEAR DOCS: If ambiguous, use best judgment and append "(inferred — verify)" to description. If docs mention version requirements, note them (e.g. "requires curl 7.66+"). Never invent flags not in the docs.

12. MIN/MAX: Only valid for integer and float types. Use when docs specify a valid range (e.g., -T 0-5, --quality 1-31). When set, the GUI constrains the spinner. Default null (unconstrained). If both are set, min must be <= max.

13. DEPRECATED: Set to a string message when docs mark a flag as deprecated, obsolete, or removed in newer versions. Include the recommended replacement if docs mention one. The GUI still allows the flag but shows a visual warning. Default null.

14. DANGEROUS: Set to true for flags that bypass safety checks (--force), can destroy data (--overwrite, --remove-all), disable security features, or have other irreversible effects. Default false.

15. PASSWORD: Use for flags that accept passwords, API keys, tokens, or other sensitive strings. The GUI masks the input. Only use when the value is clearly sensitive — not for regular strings.

16. SCRIPT BINARIES: If the tool is a script file (.py, .sh, .bat, .cmd) rather than a system binary, use the script's filename as the binary field value (e.g. "binary": "rayhunter.py"). The user configures Scaffold's custom PATH to include the directory containing the script. Do not use absolute paths to script files — use bare filenames just like system binaries. On Windows, if the script is a .ps1 PowerShell script, use "binary": "powershell" and add the script path as the first positional argument with "default" set to the script filename.

17. RELEASE BINARIES: If the tool is typically downloaded as a release binary (e.g. rayhunter, self-compiled tools) rather than installed via a package manager, note this alongside your JSON output so the user knows they may need to add the binary to their PATH or set an absolute path in the `binary` field.

=== COMMAND ASSEMBLY ===

Without subcommands: <binary> [flags] [positional args]
With subcommands: <binary> [global flags] <subcommand> [subcommand flags] [positional args]
With multi-level subcommands: <binary> [global flags] <sub1> <sub2> [subcommand flags] [positional args]
